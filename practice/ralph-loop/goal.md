# 목표 명세 (Goal Specification)

> 5장 로드맵 2단계: "검증 가능한 목표 명세" 실습.
> 루프 엔지니어링의 제1원칙 — **LLM에는 '완료' 개념이 없다. 게이트가 곧 계약이다.**

## 목표 (검증 가능)

`sample/` 미니 프로젝트의 **모든 테스트가 통과**할 때까지 에이전트 루프를 돌린다.

## 게이트 (Gate) — 종료 조건의 유일한 기준

| 항목 | 정의 |
|------|------|
| 명령 | `node --test --test-reporter=tap sample/calc.test.js` |
| 성공 | 종료 코드 `0` (실패 테스트 0개) |
| 실패 | 종료 코드 ≠ `0` → 루프 계속 |

"코드가 좋아 보인다", "거의 다 됐다" 같은 주관 판단은 게이트가 아니다.
**게이트는 기계가 판정한다.** (2장: 검증자 분리 — 작업자가 자기 완료를 선언하지 않는다)

## 가드레일 (수치 명세)

| 가드레일 | 값 | 위반 시 |
|---------|-----|--------|
| 반복 상한 (`MAX_ITER`) | 5 | `exit 1` — 상한 도달 실패 |
| 무진행 감지 (`STALL_LIMIT`) | 연속 2회 실패 수 동일 | `exit 2` — 무진행 중단 |

## 외부 메모리

- `progress.md` — 매 반복의 실패 수와 조치를 기록한다.
- 에이전트는 반복마다 **새 프로세스(새 컨텍스트)** 로 호출되며,
  상태는 오직 파일시스템(소스 코드 + progress.md)에서만 읽는다. (3장: Ralph Loop 원리)

## 초기 상태 (의도된 결함)

`sample/calc.js`에 버그 2개가 심어져 있다:

1. `add(a, b)`가 `a - b`를 반환 (오연산)
2. `divide(a, b)`가 `b === 0`을 검사하지 않음 (0 나누기 미처리)

→ 초기 게이트 실행 시 **정확히 2개 테스트 실패**여야 한다.
