# 실습: ralph식 루프 직접 구현

> [5장 로드맵](../../docs/loop-engineering/05-리스크와-실천-가이드.md) 2~3단계 실습.
> "검증 가능한 목표 명세 → ralph식 단순 반복 루프 구성"을 80줄짜리 셸 스크립트로 직접 만들어본다.

## 무엇을 만들었나

버그 2개가 심어진 미니 프로젝트(`sample/`)를, 에이전트 루프가 **사람 개입 없이**
테스트가 전부 통과할 때까지 고치게 한다.

```
./ralph.sh 실행
  ├─ iter 1: 게이트 실행 → 실패 2개 → 에이전트 호출(새 컨텍스트) → add() 수정
  ├─ iter 2: 게이트 실행 → 실패 1개 → 에이전트 호출(새 컨텍스트) → divide() 수정
  └─ iter 3: 게이트 실행 → 실패 0개 → ✅ GATE PASSED, exit 0
```

## 빠른 시작

```bash
cd practice/ralph-loop

./ralph.sh        # mock 에이전트로 루프 실행 (2회 반복 후 수렴)
cat progress.md   # 외부 메모리에 남은 기록 확인
./reset.sh        # 버그 상태로 복원 → 다시 실습 가능
```

가드레일 동작도 직접 볼 수 있다:

```bash
AGENT_CMD=/usr/bin/true ./ralph.sh                  # 아무것도 안 하는 에이전트 → 무진행 감지 (exit 2)
STALL_LIMIT=99 MAX_ITER=2 AGENT_CMD=/usr/bin/true ./ralph.sh  # 반복 상한 도달 (exit 1)
./reset.sh                                          # 실습 후 복원
```

## 루프 엔지니어링 개념 → 코드 매핑

> **루프 = 트리거 + 검증 가능한 목표**

| 개념 | 이 실습에서의 구현 | ralph.sh 위치 |
|------|-------------------|--------------|
| 트리거 | `for` 루프가 에이전트를 반복 호출 | ③ |
| 검증 가능한 목표 (게이트) | `node --test` 종료 코드 0 — 기계가 판정 | `fail_count()`, ① |
| 새 컨텍스트 (Ralph 원리) | 에이전트를 매번 **새 프로세스**로 실행, 대화 기록 없음 | ③ |
| 외부 메모리 | `progress.md`에 반복별 기록 추가 | ④ |
| 반복 상한 | `MAX_ITER` 도달 시 exit 1 | ⑤ |
| 무진행 감지 | 실패 수가 `STALL_LIMIT`회 연속 동일하면 exit 2 | ② |
| 검증자 분리 | 에이전트(작업자)가 아닌 테스트 러너(평가자)가 완료 판정 | `fail_count()` |

## 실제 에이전트로 업그레이드 (로드맵 3단계 →)

mock 대신 진짜 코딩 에이전트를 꽂으면 그대로 실전 ralph 루프가 된다:

```bash
AGENT_CMD='claude -p "practice/ralph-loop/goal.md를 읽고, sample/의 실패하는 테스트를 정확히 1개만 고쳐라. 수정 후 즉시 종료하라."' ./ralph.sh
```

핵심은 **루프 구조가 바뀌지 않는다**는 점 — 에이전트가 mock이든 Claude든,
게이트·가드레일·외부 메모리는 동일하게 동작한다. 에이전트 품질 = 루프 품질이 아니라,
**루프가 에이전트 품질의 하한선을 보증한다.**

## 종료 코드 계약

| exit | 의미 |
|------|------|
| `0` | ✅ 게이트 통과 (목표 달성) |
| `1` | ⛔ 반복 상한 도달 |
| `2` | ⛔ 무진행 감지 |
