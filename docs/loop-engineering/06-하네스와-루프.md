# 6장 (부록). 하네스 엔지니어링과 루프 엔지니어링

둘은 자주 함께 언급되지만 **설계 대상이 다른, 레이어가 다른 개념**이다.

## 6.1 한 줄 구분

- **하네스 엔지니어링 (Harness Engineering)**: 에이전트가 **어떤 환경 안에서** 도는가 — 모델을 감싸는 "작업장"을 설계.
- **루프 엔지니어링 (Loop Engineering)**: 에이전트가 **언제, 왜, 언제까지** 도는가 — 작업장 위에서 돌아가는 "운영 체계"를 설계.

비유: 하네스가 **공장 설비**(작업대, 공구함, 안전 펜스, CCTV)라면, 루프는 **생산 스케줄과 품질 검수 프로세스**(언제 라인을 돌리고, 무엇이 완성 기준이고, 불량이면 어떻게 되돌리는가)다.

Osmani의 정리: 루프 엔지니어링은 하네스 엔지니어링 **위에 얹히는(sits above)** 개념이다. 하네스 엔지니어링은 "컨텍스트 엔지니어링의 다음 진화"로 불린다.

## 6.2 스택으로 보기

```
┌─────────────────────────────────────────────┐
│  루프 엔지니어링          "언제·왜·언제까지"     │
│  트리거 · 목표 · 종료 게이트 · 평가자 분리 ·     │
│  외부 메모리 · 반복/예산 상한                   │
├─────────────────────────────────────────────┤
│  하네스 엔지니어링         "무엇을 갖고·어디까지"  │
│  도구 표면 · 샌드박스/권한 · 컨텍스트 정책 ·     │
│  모델 라우팅 · 관측성                          │
├─────────────────────────────────────────────┤
│  모델 (LLM)               "추론 능력"          │
└─────────────────────────────────────────────┘
```

## 6.3 설계 질문으로 비교

| | 하네스 엔지니어링 | 루프 엔지니어링 |
|---|---|---|
| **핵심 질문** | "에이전트에게 무엇을 쥐여주고 어디까지 허용할까?" | "누가 일을 발견하고, 언제 멈추고, 누가 검증할까?" |
| **설계 대상** | 도구 표면(tool surface), 샌드박스/권한, 컨텍스트 공급 정책, 모델 라우팅, 관측성(로그·추적) | 트리거(`/loop`, cron), 검증 가능한 목표, 종료 게이트, 평가자 분리, 외부 메모리, 반복·예산 상한 |
| **실패하면** | 엉뚱한 도구 사용, 위험한 행동, 컨텍스트 오염 | 무한 루프, 목표 표류, 진행 환각, 토큰 폭발 |
| **시간 단위** | 한 번의 실행(run) **내부**의 품질 | 실행과 실행 **사이**를 잇는 연속성 |
| **계보** | 컨텍스트 엔지니어링의 다음 진화 | 프롬프트→컨텍스트→루프로 이어지는 다음 단계 |

## 6.4 경계는 칼같지 않다 — 7-Plane 프레임워크

하네스를 7개 평면(plane)으로 정의하는 프레임워크가 있다:

1. **루프 정책 (loop policy)** ← 주목
2. 도구 표면 (tool surface)
3. 컨텍스트 (context)
4. 샌드박스 (sandbox)
5. 멀티 에이전트 라우팅 (multi-agent routing)
6. 관측성 (observability)
7. 모델 라우팅 (model routing)

"루프 정책"이 하네스의 한 평면으로 **포함**되어 있다는 점이 핵심이다. 즉:

> 하네스는 루프가 돌 수 있는 **능력과 한계**를 정의하고,
> 루프 엔지니어링은 그 능력을 **목표 달성 시스템으로 조립**한다.

## 6.5 구성요소를 레이어에 배치해보기 (4장과 연결)

4장의 5대 구성요소를 두 레이어에 나눠 보면:

| 구성요소 | 주 레이어 | 이유 |
|---------|----------|------|
| 자동화 (`/loop`, `/goal`) | **루프** | 트리거와 종료가 본질 |
| 서브에이전트 (검증 분리) | **루프** | "누가 검증하는가"의 문제 |
| 외부 메모리 | **루프** | 실행 사이의 연속성 |
| 스킬 (SKILL.md) | **하네스** | 컨텍스트 공급 정책 |
| 플러그인 & 커넥터 (MCP) | **하네스** | 도구 표면 확장 |
| 워크트리 | **하네스** | 샌드박스/격리 |

## 6.6 결론

- 둘은 경쟁 개념이 아니라 **스택의 상하층**이다.
- 같은 하네스 위에서도 **루프 설계에 따라 결과가 갈린다.**
- 좋은 루프도 부실한 하네스(도구 부족, 관측 불가, 격리 없음) 위에서는 **신뢰할 수 없다.**
