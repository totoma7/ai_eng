# 9장 (심화). 에이전트 Evals 설계 — 게이트의 품질이 곧 루프의 품질

> 2장의 원칙 — "검증 게이트가 신뢰의 유일한 원천" — 을 체계적인 평가 프로그램으로 확장한다.
> [practice/ralph-loop](../../practice/ralph-loop/README.md)의 `node --test` 게이트가 이 장의 최소 사례다.

## 9.1 왜 Evals가 전략 인프라가 됐나

- Promptfoo가 OpenAI에 인수 (2026.3, $86M), Braintrust가 $800M 밸류에이션 — 평가 도구가 **전략적 인프라 카테고리**로 격상.
- 체계적 평가 프로그램(인시던트 분류 체계 + 골든 데이터셋 + 채점기 + CI 게이트)을 갖춘 팀이 **구조적 품질 우위**를 가진다.

## 9.2 커스텀 Eval 설계 5단계

### Step 1 — 실패 모드 분류 체계 (Failure Taxonomy)
**실제 프로덕션 인시던트**에서 출발한다: 도구 오선택, 환각 출력, 무한 루프, 데이터 유출.
가상의 시나리오를 지어내지 말고 **실제 실패 사례 10~20건**을 문서화 → 이것이 평가 카테고리가 된다.
(4장의 실패 모드 도감이 출발점)

### Step 2 — 골든 데이터셋 구축
- 실패 모드당 **50~200개의 수작업 라벨링** 예시. 합성 생성이 아니라 도메인 전문성으로.
- **git으로 버전 관리** — 일급 엔지니어링 자산으로 취급.
- 양보다 질.

### Step 3 — 채점기 믹스: 60 / 30 / 10
| 비중 | 채점기 유형 | 예시 |
|------|-----------|------|
| 60% | **결정적 채점기** | 정확 일치, 정규식, 스키마 검증, 지연 임계값 — ralph-loop의 테스트 게이트가 여기 |
| 30% | **LLM 판정자** (LLM-as-judge) | G-Eval, DAG 메트릭 — 자연어 품질, 적합성 |
| 10% | **사람** | 모호한 케이스 |

> "LLM 판정자에만 의존하지 마라 — 에이전트의 확률성 위에 **채점기의 확률성**을 얹는 꼴이다."

### Step 4 — 베이스라인 + 합격선
- 합격선을 정하기 **전에** 현재 프로덕션 에이전트를 먼저 채점한다.
- 절대 점수가 아니라 **베이스라인 대비 회귀**를 차단한다.
- 예: "도구 선택: 베이스라인 88%, 목표 95% — 베이스라인 미만이면 배포 차단."

### Step 5 — CI/CD 회귀 게이트
- 평가 실행을 PR 워크플로우에 연결. 평가 실패 = 머지 차단.
- 합격선 ±2%p 경계 케이스는 알림(Slack 등)으로.
- 루프 엔지니어링 관점: **이것이 곧 "사람 없이 도는 루프"의 게이트를 조직 레벨로 올린 것**이다.

## 9.3 하네스 효과 — 리더보드를 믿지 마라

> "동일한 모델 가중치가 하네스에 따라 SWE-Bench Verified에서 **10~20점** 차이를 낸다."

벤더 리더보드 점수는 평가 스캐폴딩·도구 접근·재시도 집계 방식이 달라 **상호 비교 불가**.
실무 결론: **자기 워크로드를 대표하는 골든 데이터셋 벤치마크를 직접 만들어라.**
공개 점수는 "그들의 하네스 안의 모델"을 말해줄 뿐, 당신의 하네스를 말해주지 않는다.

## 9.4 하네스 자체를 평가하라 (Harness Evals)

표준 벤치마크가 안 잡는 실패 표면 — 배포 전 필수 점검:

| 평가 항목 | 질문 |
|----------|------|
| **인젝션 저항성** | 사용자 프롬프트가 시스템 지시를 덮어쓸 수 있는가? |
| **타임아웃 복원력** | 도구가 멈췄을 때 하네스가 멈추는가, 같이 멈추는가? |
| **과잉 도구 요청** (over-tooling) | 에이전트가 불필요한 도구를 요청하는가? |

### 오픈소스
| 프레임워크 | 특징 |

벤더 중립성 메모: Promptfoo의 OpenAI 인수로 비-OpenAI 모델 커버리지 격차가 **구조적 리스크**가 됨 → Inspect AI나 DeepEval을 2차 의견 레이어로.

## 9.6 ralph-loop 실습과의 연결

실습의 게이트를 이 장의 언어로 업그레이드하는 경로:

1. 현재: 결정적 채점기 100% (`node --test` 종료 코드) — Step 3의 60% 영역
2. 다음: 코드 품질 LLM 판정자 추가 — "수정이 테스트만 통과시키는 꼼수인가, 진짜 수정인가?" (30% 영역)
3. 다음: 실패 사례를 골든 데이터셋으로 축적 → 새 mock 버그 추가 시 회귀 게이트로
