02

에이전트 루프의 해부학

에이전트는 과업을 받으면 세 단계를 순환한다. 단계는 칼같이 나뉘지 않고 섞이며, 무엇이 필요한지는 직전 단계에서 배운 것으로 에이전트가 결정한다.

① 수집GATHER CONTEXT파일 검색 · 코드 읽기 · 기록 확인
② 실행TAKE ACTION코드 수정 · 명령 실행 · API 호출
③ 검증VERIFY WORK테스트 · 타입 체크 · 결과 확인
실패 시오류 → 새 컨텍스트
피드백 없는 루프는 자기 동의일 뿐

밀어낼(push back) 것이 없으면 에이전트는 자신에게 동의한다. 테스트·타입·리뷰 게이트가 신뢰의 유일한 원천.

LLM에는 '완료' 개념이 없다

검증 가능한 종료 조건이 없는 목표는 토큰 소각로가 된다. 목표가 추상적일수록 루프는 비싸고 예측 불가능해진다.

종료는 태스크가 아니라 게이트로

"모든 태스크 완료"가 아니라 "모든 게이트 충족"이 계약. 태스크는 비계일 뿐이다. tests ✓ coverage ✓ review ✓

탐색하고 구현한다. 그러나 자기 완료를 선언할 수 없다.

"done?" ⇄ 별도 채점
EVALUATOR 평가자 모델

매 턴 끝에 목표 조건을 검사. 통과할 때만 루프를 멈춘다. (/goal의 구조)

검증자 분리(Verification Split) — 자기 숙제를 자기가 채점하지 않게 하라. "done"이 신뢰를 얻는 유일한 구조.

04

5대 구성요소 + 척추

Addy Osmani가 정리한 루프 엔지니어링의 빌딩 블록.

01 HEARTBEAT

자동화 (Automations)

주기적 스케줄 태스크가 일을 발견하고 트리아지. 사람이 확인하러 가는 게 아니라 작업이 인박스로 올라온다. /loop · /goal

02 ISOLATION

워크트리 (Worktrees)

git worktree로 작업 디렉토리를 분리. 여러 에이전트가 파일 충돌 없이 동시 작업한다.

03 KNOWLEDGE

스킬 (Skills)

SKILL.md로 묶인 재사용 지시 패키지. 컨벤션·빌드 절차를 인코딩해 매 실행마다 같은 설명을 반복할 필요를 제거.

04 INTEGRATION

플러그인 & 커넥터

MCP 기반으로 이슈 트래커·DB·Slack과 연동. 루프를 '제안자'에서 PR 열고 티켓 갱신하는 실제 참여자로 전환.

05 SEPARATION

서브에이전트

구현 에이전트와 검증 에이전트를 분리. "자기 숙제 자기 채점" 방지. 역할별로 다른 모델·노력 수준 배정 가능.

SPINE — 척추

외부 메모리

대화 밖에 존재하는 마크다운 파일이나 보드. 모델은 실행 사이에 잊어버리므로, 영속 상태가 완료된 일과 다음 단계를 기억한다. 다음 실행은 이전 상태에서 이어간다.

05

가드레일 vs 실패 모드

"목표는 자율성을 없애는 게 아니라 경계 짓는(bound) 것이다."

GUARDRAILS — 필수 장치

  • 반복 상한 — 최대 사이클 수 하드 캡
  • 토큰/달러 예산 — 실행 전에 설정
  • 무진행 감지 — 상태가 안 변하면 탈출
  • 자동 검증 기준 — 테스트·커버리지·타입 체크
  • 타임아웃 — 도구 호출 단위 + 태스크 단위
  • 서킷 브레이커 — 도구 재시도 폭주 차단
  • 휴먼 체크포인트 — 머지·배포·삭제 전 사람 승인

FAILURE MODES — 실패 도감

  • 무한 루프 — 멈출 조건 없이 계속 다듬음
  • 목표 표류 — 모호한 명세 → 곁가지 추구
  • 컨텍스트 오버플로 — 긴 세션이 추론 품질 저하
  • 침묵 실패 — 도구는 도는데 진전은 없음
  • 토큰 폭발 — 사례: 5분에 중복 호출 400회
  • 진행 환각 — "done"은 주장이지 증명이 아니다

06

세 가지 본질적 리스크

루프가 잘 돌수록 위험해지는 것들.

RISK 01

검증은 끝까지 당신 책임

방치된 루프는 방치된 채로 실수한다. 자동 게이트가 검사하지 않는 영역은 결국 사람이 본다.

RISK 02

이해 부채 (Comprehension Debt)

코드가 이해 속도보다 빠르게 출하되면 '존재하는 것'과 '파악한 것'의 격차가 벌어진다. 루프 출력을 읽는 것은 타협 불가.

RISK 03

인지적 항복 (Cognitive Surrender)

같은 루프를 가진 두 엔지니어가 정반대 결과를 얻는다 — 한 명은 깊이 이해하고, 한 명은 생각을 회피한다. 루프는 어느 쪽인지 모른다.

"루프를 만들어라. 단, 엔지니어로 남을 사람처럼 만들어라.
'실행 버튼만 누르는 사람'이 아니라."

— Addy Osmani, Loop Engineering (2026)

07

부록 — 하네스 vs 루프

하네스가 공장 설비(작업대·공구함·안전 펜스·CCTV)라면, 루프는 생산 스케줄과 품질 검수 프로세스다. 경쟁 개념이 아니라 스택의 상하층.

루프 엔지니어링WHEN · WHY · UNTIL

"누가 일을 발견하고, 언제 멈추고, 누가 검증할까?"

트리거 /loop·cron검증 가능한 목표 종료 게이트평가자 분리 외부 메모리반복·예산 상한
하네스 엔지니어링WHAT · HOW FAR

"에이전트에게 무엇을 쥐여주고 어디까지 허용할까?"

도구 표면샌드박스·권한 컨텍스트 정책 (스킬)커넥터 MCP 워크트리 격리관측성모델 라우팅
모델 (LLM)REASONING

"추론 능력" — 같은 모델이라도 위 두 레이어의 설계가 결과를 가른다.

하네스는 루프가 돌 수 있는 능력과 한계를 정의하고,
루프 엔지니어링은 그 능력을 목표 달성 시스템으로 조립한다.
