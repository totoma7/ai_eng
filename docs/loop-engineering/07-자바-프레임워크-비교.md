# 7장 (부록). 진화 단계별 차이 — LangChain4j vs Spring AI 코드로 보기

> 프롬프트 → 컨텍스트 → 루프 엔지니어링의 차이를 **같은 시나리오**를
> 자바 양대 프레임워크로 구현하며 비교한다.
>
> 시나리오: **"주문 도우미"** — 사용자가 주문 상태를 묻고, 필요하면 환불까지 처리하는 어시스턴트.

## 7.1 진화 단계가 코드에서 의미하는 것

| 단계 | 설계 대상 | 코드에서 바뀌는 것 | 제어의 주체 |
|------|----------|------------------|------------|
| ① 프롬프트 엔지니어링 | 한 번의 요청문 | **문자열** (프롬프트 텍스트) | 사람 (매 호출) |
| ② 컨텍스트 엔지니어링 | 정보 공급 구조 | **입력 조립 파이프라인** (메모리·RAG) | 프레임워크 (호출 시점) |
| ③ 루프 엔지니어링 | 반복·종료 제어 | **제어 흐름** (도구 루프 + 게이트 + 가드레일) | 시스템 (목표 달성까지) |

핵심: 단계가 올라갈수록 코드의 무게중심이 **"무엇을 말할까"에서 "언제 멈출까"로** 이동한다.

## 7.2 ① 프롬프트 엔지니어링 — 단일 호출

요청문을 잘 쓰는 것이 전부인 단계. 모델은 한 번 답하고 끝난다.
상태도, 도구도, 반복도 없다.

### LangChain4j

```java
// 방식 A: 모델 직접 호출
ChatModel model = OpenAiChatModel.builder()
        .apiKey(System.getenv("OPENAI_API_KEY"))
        .modelName("gpt-4.1")
        .build();

String answer = model.chat("주문 #1234의 일반적인 배송 기간을 알려줘.");

// 방식 B: 선언적 AI Service — 프롬프트를 인터페이스에 부착
interface OrderAssistant {
    @SystemMessage("당신은 주문 도우미입니다. 정중하고 간결하게 답하세요.")
    String answer(@UserMessage String question);
}

OrderAssistant assistant = AiServices.builder(OrderAssistant.class)
        .chatModel(model)
        .build();

String answer = assistant.answer("주문 #1234의 일반적인 배송 기간을 알려줘.");
```

### Spring AI

```java
// ChatClient 플루언트 API — 요청을 체이닝으로 조립
@Service
public class OrderAssistant {

    private final ChatClient chatClient;

    public OrderAssistant(ChatClient.Builder builder) {
        this.chatClient = builder
                .defaultSystem("당신은 주문 도우미입니다. 정중하고 간결하게 답하세요.")
                .build();
    }

    public String answer(String question) {
        return chatClient.prompt()
                .user(question)
                .call()
                .content();
    }
}
```

### 이 단계의 흐름 차이

```
LangChain4j : 인터페이스 정의 → 프록시 생성 → 메서드 호출이 곧 LLM 호출
Spring AI   : 빈 주입 → prompt() 체인 조립 → call()이 곧 LLM 호출
```

- **LangChain4j**: "LLM 호출을 **자바 인터페이스**로 추상화한다." 프롬프트가 어노테이션으로 코드에 선언된다.
- **Spring AI**: "LLM 호출을 **HTTP 클라이언트처럼** 다룬다." `WebClient`를 쓰던 감각 그대로다.
- 한계(공통): 모델은 주문 #1234의 **실제 상태를 모른다**. 일반론만 답한다. → ②로.

## 7.3 ② 컨텍스트 엔지니어링 — 정보 공급 구조

"모델에게 어떤 정보를 어떤 형태로 공급할 것인가."
대화 기억(메모리)과 사내 데이터(RAG)를 **호출 시점에 자동 조립**한다.
여전히 모델은 한 턴에 한 번 답하지만, 입력의 질이 달라진다.

### LangChain4j — 빌더에 공급 장치를 부착

```java
interface OrderAssistant {
    @SystemMessage("당신은 주문 도우미입니다. 제공된 문서와 대화 맥락에 근거해 답하세요.")
    String answer(@UserMessage String question);
}

OrderAssistant assistant = AiServices.builder(OrderAssistant.class)
        .chatModel(model)
        // 대화 기억: 최근 10개 메시지를 자동으로 함께 전송
        .chatMemory(MessageWindowChatMemory.withMaxMessages(10))
        // RAG: 질문과 유사한 문서 조각을 벡터 스토어에서 찾아 프롬프트에 주입
        .contentRetriever(EmbeddingStoreContentRetriever.builder()
                .embeddingStore(embeddingStore)   // 배송 정책, FAQ 등이 임베딩되어 있음
                .embeddingModel(embeddingModel)
                .maxResults(3)
                .build())
        .build();

// 호출 코드는 ①과 동일 — 공급 구조만 두꺼워졌다
assistant.answer("주문 #1234 아직 안 왔는데 언제 와요?");
assistant.answer("그거 환불 규정은 어떻게 돼요?");   // "그거" = 메모리가 맥락 유지
```

### Spring AI — Advisor 체인에 공급 장치를 끼움

```java
ChatClient chatClient = ChatClient.builder(chatModel)
        .defaultSystem("당신은 주문 도우미입니다. 제공된 문서와 대화 맥락에 근거해 답하세요.")
        .defaultAdvisors(
                // 대화 기억: 요청 전에 히스토리를 붙이고, 응답 후 저장
                MessageChatMemoryAdvisor.builder(chatMemory).build(),
                // RAG: 질문과 유사한 문서를 벡터 스토어에서 찾아 프롬프트에 주입
                QuestionAnswerAdvisor.builder(vectorStore).build())
        .build();

String answer = chatClient.prompt()
        .advisors(a -> a.param(ChatMemory.CONVERSATION_ID, "user-42"))  // 대화 식별
        .user("주문 #1234 아직 안 왔는데 언제 와요?")
        .call()
        .content();
```

### 이 단계의 흐름 차이

```
LangChain4j : 호출 → [프록시가 메모리 로드 + 리트리버 검색 + 프롬프트 조립] → LLM → 응답
Spring AI   : 호출 → [Advisor 체인: 메모리 어드바이저 → RAG 어드바이저] → LLM → [체인 역순 통과] → 응답
```

- **LangChain4j**: 공급 장치가 **빌더 옵션**이다. AI Service 프록시가 내부에서 전부 처리한다. 간결하지만 중간 과정에 끼어들 지점이 적다.
- **Spring AI**: 공급 장치가 **체인의 노드**다. 서블릿 필터/AOP처럼 요청·응답 양방향을 통과하므로, 커스텀 Advisor를 만들어 어느 지점이든 끼어들 수 있다 (로깅, 가드, 프롬프트 변형).
- 한계(공통): 모델이 환불 **"방법을 설명"** 할 수는 있지만 **"실행"** 할 수는 없다. 행동이 없다. → ③으로.

## 7.4 ③ 루프 엔지니어링 — 도구 루프와 게이트

> **루프 = 트리거 + 검증 가능한 목표.**

이 단계에서 루프는 두 겹이다:

1. **안쪽 루프 (프레임워크 제공)** — 도구 호출 루프: 모델이 도구를 부르고 → 결과를 보고 → 다음 행동을 정하는 반복. LangChain4j의 AI Service와 Spring AI의 `ToolCallAdvisor`가 자동 관리한다.
2. **바깥 루프 (직접 설계)** — 검증 게이트·반복 상한·외부 메모리: ralph식 루프. **이것이 루프 엔지니어링의 본령**이며, 어느 프레임워크도 공짜로 주지 않는다.

### 도구 정의 (양쪽 거의 동일한 모양)

```java
// LangChain4j
class OrderTools {
    @Tool("주문 번호로 주문 상태를 조회한다")
    OrderStatus getOrderStatus(String orderId) { return orderRepository.findStatus(orderId); }

    @Tool("주문을 환불 처리하고 결과를 반환한다")
    RefundResult refund(String orderId, String reason) { return refundService.refund(orderId, reason); }
}

// Spring AI — @ToolParam으로 파라미터 설명 추가
class OrderTools {
    @Tool(description = "주문 번호로 주문 상태를 조회한다")
    OrderStatus getOrderStatus(@ToolParam(description = "주문 번호") String orderId) { ... }

    @Tool(description = "주문을 환불 처리하고 결과를 반환한다")
    RefundResult refund(String orderId, String reason) { ... }
}
```

### 안쪽 루프 — 도구 호출 반복 (프레임워크가 관리)

```java
// LangChain4j: tools()를 부착하면 AI Service가 도구 루프를 돈다
OrderAgent agent = AiServices.builder(OrderAgent.class)
        .chatModel(model)
        .chatMemory(MessageWindowChatMemory.withMaxMessages(20))
        .tools(new OrderTools())
        .build();

// 이 한 줄 뒤에서: 모델이 getOrderStatus 호출 → 결과 관찰 → refund 호출 → 최종 응답
agent.handle("주문 #1234가 3주째 안 와요. 환불해 주세요.");
```

```java
// Spring AI: ToolCallAdvisor가 어드바이저 체인 안에서 도구 루프를 돈다
String result = chatClient.prompt()
        .user("주문 #1234가 3주째 안 와요. 환불해 주세요.")
        .tools(new OrderTools())
        .call()
        .content();
```

안쪽 루프의 흐름 (양쪽 동일한 패턴, 관리 주체만 다름):

```
사용자 요청
   ↓
LLM: "getOrderStatus('1234') 호출 필요" ──→ 도구 실행 ──→ 결과를 대화에 추가
   ↓ (새 LLM 호출)
LLM: "3주 지연 확인. refund('1234', '배송 지연') 호출" ──→ 도구 실행 ──→ 결과 추가
   ↓ (새 LLM 호출)
LLM: "환불 완료되었습니다" → 도구 호출 없음 → 루프 종료, 최종 응답 반환
```

### 바깥 루프 — ralph식 게이트 루프 (직접 설계)

안쪽 루프의 종료 조건은 "모델이 더 이상 도구를 부르지 않음" — 즉 **모델의 자기 선언**이다.
2장의 원칙대로 이것을 신뢰하지 않고, **독립 검증자와 가드레일**로 감싼다.
(아래는 [practice/ralph-loop/ralph.sh](../../practice/ralph-loop/README.md)와 동일한 구조의 자바 버전 — 프레임워크 무관)

```java
public RefundOutcome handleWithGate(String request) {
    int maxIter = 3;                                   // 가드레일: 반복 상한
    List<String> progress = new ArrayList<>();         // 외부 메모리

    for (int i = 1; i <= maxIter; i++) {
        String claim = agent.handle(request);          // 안쪽 루프 실행 (작업자)

        // 게이트: 작업자의 주장이 아니라 시스템 상태로 검증 (검증자 분리)
        OrderStatus actual = orderRepository.findStatus("1234");
        if (actual.isRefunded()) {
            return RefundOutcome.success(claim, progress);   // ✅ 게이트 통과
        }

        progress.add("iter %d: 환불 미완료, 모델 주장: %s".formatted(i, claim));
        request = "이전 시도에서 환불이 완료되지 않았다. 기록: %s. 다시 시도하라."
                .formatted(progress.getLast());        // 실패를 새 컨텍스트로
    }
    return RefundOutcome.escalate(progress);           // ⛔ 상한 도달 → 사람에게 (휴먼 체크포인트)
}
```

이 짧은 바깥 루프가 ①·②와 ③을 가르는 본질이다 — **"done"은 모델의 주장이고, 게이트가 증명이다.**

## 7.5 두 프레임워크의 흐름(아키텍처) 차이 총정리

```
LangChain4j — "선언적 프록시" 모델
┌──────────────────────────────────────────────┐
│  자바 인터페이스 (프롬프트·반환 타입 선언)          │
│       ↓ AiServices.builder()                  │
│  동적 프록시 [메모리 │ RAG │ 도구 루프] 일체형     │
│       ↓                                       │
│  ChatModel                                    │
└──────────────────────────────────────────────┘
"LLM 기능을 자바 타입 시스템 안으로 끌어들인다"

Spring AI — "요청 파이프라인" 모델
┌──────────────────────────────────────────────┐
│  ChatClient.prompt() 플루언트 체인              │
│       ↓                                       │
│  Advisor 체인 (요청→ ⟶ ←응답 양방향 통과)        │
│   [메모리 Advisor]→[RAG Advisor]→[ToolCall     │
│    Advisor]→ ... 끼우고 빼고 순서 바꾸기 자유      │
│       ↓                                       │
│  ChatModel                                    │
└──────────────────────────────────────────────┘
"LLM 호출을 스프링의 횡단 관심사 파이프라인으로 다룬다"
```

| 관점 | LangChain4j | Spring AI |
|------|------------|-----------|
| 추상화 단위 | **인터페이스** (AI Service 프록시) | **요청 체인** (ChatClient + Advisor) |
| 프롬프트 위치 | 어노테이션 (`@SystemMessage`, `@UserMessage`) | 플루언트 체인 (`.system()`, `.user()`) |
| 메모리/RAG 결합 | 빌더 옵션 (일체형) | Advisor 노드 (조립형) |
| 도구 루프 관리 | AI Service 내부 | `ToolCallAdvisor` (체인의 한 노드) |
| 확장 지점 | 커스텀 RetrievalAugmentor, ToolProvider 등 | **커스텀 Advisor** — 서블릿 필터 감각 |
| 스프링 의존 | 없음 (순수 자바, 스프링 부트 스타터는 별도) | 스프링 생태계 전제 (DI, 부트, 옵저버빌리티) |
| 멘탈 모델 | "AI 기능을 가진 자바 객체" | "AI를 향한 HTTP 클라이언트 + 미들웨어" |
| 바깥 루프(게이트) | ❌ 직접 구현 | ❌ 직접 구현 |

**선택 가이드**: 스프링 부트 기반 서비스에 자연스럽게 녹이려면 Spring AI,
프레임워크 중립적이거나 선언적 스타일을 선호하면 LangChain4j.
**바깥 루프(게이트·가드레일·외부 메모리)는 어느 쪽이든 직접 설계해야 한다** — 그것이 루프 엔지니어링이다.

### 더 가면 (그래프 오케스트레이션)

바깥 루프가 복잡해지면(분기·병렬·멀티 에이전트) 상태 그래프 프레임워크로 넘어간다:

- **LangGraph4j** — LangChain4j/Spring AI 양쪽과 연동되는 상태 그래프 (LangGraph의 자바판)
- **Spring AI Alibaba Graph** — Spring AI 위의 그래프 기반 멀티 에이전트 오케스트레이션

이들은 4장의 "오케스트레이션 루프"를 코드 레벨에서 지원하는 도구다.

## 7.6 요약

1. ①→②→③은 **문자열 → 입력 조립 → 제어 흐름**으로 설계 대상이 이동하는 것.
2. 도구 호출 루프(안쪽)는 두 프레임워크 모두 자동 관리한다 — 여기까지는 프레임워크의 영역.
3. **루프 엔지니어링의 본령은 바깥 루프** — 게이트, 반복 상한, 외부 메모리, 휴먼 체크포인트. 어느 프레임워크도 대신해 주지 않는다.
4. LangChain4j는 **선언적 프록시**, Spring AI는 **Advisor 파이프라인** — 같은 능력, 다른 조립 철학.
