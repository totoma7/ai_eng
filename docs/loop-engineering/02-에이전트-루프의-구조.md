# 2장. 에이전트 루프의 구조 (해부학)

루프 엔지니어링을 하려면 먼저 **에이전트 루프 자체**가 어떻게 도는지 이해해야 한다.
두 가지 표준 모형이 있다: 실무형 3단계 모형과 일반화된 5단계 모형.

## 2.1 3단계 모형 — 수집 → 실행 → 검증

Claude Code 공식 문서가 설명하는 모형. 에이전트는 과업을 받으면 세 단계를 순환한다.

```
┌────────────────────────────────────────────┐
│  ① 컨텍스트 수집 (Gather Context)            │
│     파일 검색, 코드 읽기, 과거 기록 확인        │
│              ↓                              │
│  ② 행동 실행 (Take Action)                   │
│     코드 수정, 명령 실행, API 호출             │
│              ↓                              │
│  ③ 작업 검증 (Verify Work)                   │
│     테스트 실행, 타입 체크, 결과 확인           │
│              ↓                              │
│  검증 실패 → 오류 정보를 새 컨텍스트로 ①로 복귀  │
│  검증 성공 → 종료                             │
└────────────────────────────────────────────┘
```

중요한 특징:

- 단계는 칼같이 나뉘지 않고 **섞여서** 진행된다. 도구는 전 단계에서 쓰인다.
- 루프는 과업에 따라 **적응**한다: 질문이면 ①만, 버그 수정이면 ①②③을 여러 번, 리팩터링이면 ③이 집중적으로.
- 각 단계에서 무엇이 필요한지는 **직전 단계에서 배운 것**을 바탕으로 에이전트가 결정한다.

## 2.2 5단계 모형 — Perceive → Reason → Plan → Act → Observe

더 일반화된 학술/아키텍처 모형.

| 단계 | 역할 |
|------|------|
| **Perceive (지각)** | 사용자 목표, 도구 결과, API 응답, 오류를 입력으로 받음 |
| **Reason (추론)** | 입력 분석, 현재 지식 평가, 격차 식별, 선택지 비교 |
| **Plan (계획)** | 다음 행동 선택 (단순 루프: 한 스텝 / 복잡한 구조: 전체 작업 분해) |
| **Act (행동)** | 도구 호출, 파일 쓰기, 코드 실행, DB 쿼리 |
| **Observe (관찰)** | 결과 수신 후 내부 모델 갱신. 성공이면 전진, 실패면 재추론 |

매 반복(iteration)마다: 컨텍스트 조립 → LLM이 추론하여 행동 선택 → 실행 → 결과 관찰 → 다음 반복에 피드백. **과업 완료 또는 정의된 종료 조건**에 도달할 때까지 반복.

## 2.3 루프를 신뢰할 수 있게 만드는 것: 피드백

> "밀어낼(push back) 것이 없는 루프는 에이전트가 자기 자신에게 동의하는 것에 불과하다."

루프가 신뢰를 얻는 유일한 방법은 **외부 피드백 신호**다:

- 테스트 통과/실패
- 타입 체크, 린트, 빌드
- 리뷰 게이트 (별도 리뷰어 에이전트 또는 사람)
- 실행 결과 관찰 (스크린샷, 로그, 벤치마크)

핵심 원리 — **검증자 분리(Verification Split)**:
작업을 수행한 에이전트가 아닌 **별도의 평가자 모델**이 목표 충족 여부를 채점해야 "done" 신호를 신뢰할 수 있다. ("자기 숙제를 자기가 채점하지 않게 하라.") Claude Code의 `/goal`이 이 구조를 네이티브로 구현 — 매 턴 끝에 가벼운 별도 모델이 목표 조건을 검사하고, 통과할 때만 루프를 멈춘다.

## 2.4 잘 설계된 루프의 5요소

1. **검증 가능한 종료 조건이 있는 구체적 목표** — LLM에는 '완료'라는 내장 개념이 없다. 목표가 추상적일수록 루프는 비싸고 예측 불가능해진다.
2. **유용한 도구 집합** — 모델은 추론하고, 도구는 행동한다.
3. **컨텍스트 관리** — 토큰 오버플로 방지. 긴 세션은 추론 품질을 떨어뜨린다.
4. **명시적 실패 탈출구** — 무한 루프 방지 (반복 상한, 무진행 감지).
5. **오류 처리** — 도구 실패를 다음 반복의 컨텍스트로 변환.

추가 실무 원칙:

- **종료는 태스크 목록이 아니라 '게이트'로**: "모든 태스크 완료"가 아니라 "모든 게이트(테스트, 커버리지, 리뷰) 충족"이 계약이다. 태스크는 비계(飛階)일 뿐.
- **커밋 크기 단위로 변경 유지**: 리뷰어가 '검증 모드'를 유지할 수 있게. 변경이 크면 리뷰가 '그럴듯함 확인 모드'로 전락한다.
- **사람도 루프의 일부**: 언제든 개입해 방향을 틀거나 컨텍스트를 보탤 수 있어야 한다.
