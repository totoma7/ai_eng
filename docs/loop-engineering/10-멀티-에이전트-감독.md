# 10장 (심화). 멀티 에이전트 감독 — 스웜의 후퇴, 수퍼바이저의 부상

> 3장에서 "단일 루프 다음 레이어"로 예고했던 주제. 2026년 중반, 방향이 명확해졌다 —
> **자율 스웜이 아니라 경계 지어진 결정론적 워크플로우**다.

## 10.1 스웜의 후퇴

> "고도로 자율적인 스웜은 **깨지기 쉽고, 감당 못 하게 비싸고, 프로덕션에서 디버깅이 거의 불가능**하다." — 운영자들의 일관된 보고

이론적 "무제약 멀티 에이전트 메시"에서 기업들이 철수하는 이유:

- 3장의 비용 데이터 재확인: 멀티 에이전트는 토큰 **~15배**
- 창발적(emergent) 조정은 재현 불가 → 인시던트 분석 불가
- 4장의 실패 모드(목표 표류, 침묵 실패)가 에이전트 수만큼 곱해짐

## 10.2 수퍼바이저 패턴 (Supervisor Pattern)

프로덕션 표준으로 수렴한 구조:

```
            ┌─────────────────────┐
            │  수퍼바이저 (단일,     │ ← 잘 범위 지어진 주 에이전트
            │  좁은 범위)           │
            └──────────┬──────────┘
        명시적 위임 (결정론적 라우팅)
     ┌────────────┼────────────┬────────────┐
     ▼            ▼            ▼            ▼
 ┌────────┐  ┌────────┐  ┌────────┐  ┌──────────┐
 │ 플래너  │  │ 실행자  │  │ 검증자  │  │ 컴플라이언스│
 └────────┘  └────────┘  └────────┘  └──────────┘
```

원칙:

1. **단일하고 범위가 좁은 주 에이전트** — 메시가 아니라 트리
2. **명시적 위임** — 전문 에이전트(플래너·실행자·검증자·컴플라이언스 리뷰어)를 **래핑된 함수 도구**로 호출. 메인 루프에 직접 노출하지 않는다.
3. **결정론적 라우팅** — "에이전트들이 알아서 조율"이 아니라 코드가 정한 경로

루프 엔지니어링과의 관계: 수퍼바이저 패턴은 4장 오케스트레이션 루프의 **에이전트 토폴로지**다. 루프(언제·왜)가 바깥, 수퍼바이저(누가·무엇을)가 안쪽.

## 10.3 Plan–Execute–Verify (PEV) 루프와 단계 게이트

**오류 복리화 폭포(Compounding Error Cascade)** 가 수학적 근거다:

> 단계 정확도 85%라도 검증 없이 10단계를 지나면: 0.85¹⁰ ≈ **20%**

그래서 단계마다 강제 게이트를 둔다 (선택적 품질 검사가 아니라 **의무적 단계 게이팅**):

| 단계 | 역할 |
|------|------|
| **Plan** | 경계 지어진 결정론적 실행 계획 생성 |
| **Execute** | 전문 에이전트들이 증거 수집·결정 합성 |
| **Verify** | **독립** 에이전트가 증거 완전성·롤백 명확성·안전성 검증 |
| **Revise** | 검증 실패 시 피드백과 함께 실행으로 복귀 |

2장의 검증자 분리가 멀티 에이전트 스케일로 확장된 형태다. ralph-loop 실습의 다음 단계(작업자/리뷰어 분리)가 바로 PEV의 최소 구현이다.

## 10.4 Reasoning Sandwich — 티어드 모델 배치

비용과 정확도를 동시에 잡는 표준 배치:

```
계획 (고성능 모델)  ─┐
                    ├─ 중간 작업 (저비용 모델 여러 개)
검증 (고성능 모델)  ─┘
```

고성능 모델이 양 끝(계획·검증)을 잡아야 오류 누적이 차단된다 — 수학적 필연.
(8장의 멀티 티어 라우팅, OMC의 haiku/sonnet/opus 라우팅이 같은 원리)

## 10.5 Human-on-the-Loop 서킷 브레이커

"in-the-loop"(매 단계 개입)가 아니라 **"on-the-loop"(감독 + 고위험 지점만 개입)**:

| 장치 | 동작 |
|------|------|
| 도구 수준 승인 | 특정 도구 호출(예: 변경 요청 발행)이 위험 임계값 기준으로 사람 리뷰 트리거 |
| Needs-Approval 함수 | 도구가 위험도·프로덕션 영향·예산 상태로 **승인 필요 여부를 동적으로 판단** |
| 승인 게이트 | 에이전트 실행을 **동기적으로 중단**하고 운영자 결정 대기 |

4장 가드레일 표의 "휴먼 체크포인트"가 멀티 에이전트에선 **하드 서킷 브레이커**로 격상된다.

## 10.6 A2A 프로토콜 — 수평 표준

프레임워크가 다른 에이전트들 간의 발견·위임을 표준화하는 **Agent-to-Agent(A2A) 프로토콜**.
암호학적 "Agent Card"로 신원·능력을 교환한다. MCP가 수직(에이전트↔도구)이라면 A2A는 수평(에이전트↔에이전트). (8장 참조)

## 10.7 요약

1. 스웜 → 수퍼바이저: **자율성의 낭만에서 결정론적 위임으로.**
2. PEV + 단계 게이트: 검증 없는 다단계는 수학적으로 실패한다 (0.85¹⁰ ≈ 20%).
3. Reasoning Sandwich: 비싼 모델은 양 끝에.
4. 사람은 루프 **위에**(on-the-loop) — 고위험 지점의 서킷 브레이커로.
