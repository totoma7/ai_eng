# 1장. 개념과 배경

## 1.1 루프 엔지니어링이란

**루프 엔지니어링(Loop Engineering)** 은 사람이 AI 에이전트에게 직접 프롬프트를 입력하는 대신,
**스스로 일을 발견하고 → 위임하고 → 결과를 검증하고 → 다음 단계를 결정하는 시스템(루프)을 설계**하는 실천법이다.

Addy Osmani(Google)의 표현:

> "루프 엔지니어링은 '에이전트에게 프롬프트하는 사람'으로서의 자신을 대체하는 것이다.
> 그 일을 대신하는 시스템을 당신이 설계한다."

핵심 구분:

- **자동화(Automation)**: 정해진 단계의 나열을 실행한다. 분기는 하드코딩되어 있다.
- **루프(Loop)**: 내부에 **의사결정**이 있다. 매 턴마다 모델이 다음 행동을 고르고, 목표 달성 여부를 스스로 평가한다.
- 한 줄 정의: **루프 = "cron + 의사결정자"**, 또는 **"트리거 + 검증 가능한 목표"**.

## 1.3 엔지니어링 패러다임의 이동

레버리지 포인트(개입 지점)가 단계적으로 위로 올라가는 흐름으로 이해하면 좋다.

```
프롬프트 엔지니어링 (2022~)
  → "한 번의 요청을 어떻게 잘 쓸 것인가"
컨텍스트 엔지니어링 (2024~)
  → "모델에게 어떤 정보를 어떤 형태로 공급할 것인가"
루프 엔지니어링 (2026~)
  → "프롬프트와 컨텍스트 공급 자체를 수행하는 시스템을 어떻게 설계할 것인가"
```

- 프롬프트 엔지니어링: 사람이 에이전트의 손을 **한 턴씩 잡고** 이끈다.
- 루프 엔지니어링: **스스로 지탱되는 시스템**을 설계한다. 설계 대상이 개별 프롬프트에서 **오케스트레이션 구조**로 이동.
- 인접 개념인 **하네스 엔지니어링(Harness Engineering)** 은 모델을 감싸는 환경(도구 표면, 샌드박스, 컨텍스트 정책, 관측성 등)을 설계하는 것으로, 루프 엔지니어링의 하부 레이어로 볼 수 있다.

## 1.4 왜 지금인가

1. **모델 능력의 임계점 도달**: 장시간 자율 작업이 실제로 가능해짐. (한 실험: 25시간 무중단, 1,300만 토큰, 3만 줄 코드 산출)
2. **도구 성숙**: Claude Code/Codex 등에 `/loop`(주기 실행), `/goal`(조건 충족까지 실행) 같은 네이티브 루프 기능 탑재.
3. **병렬화**: git worktree 기반 격리로 5~10개 세션을 동시에 돌리는 워크플로우 보편화.

## 1.5 주의: 과장과 현실

- Osmani 본인도 "아직 초기이며 회의적인 부분이 있다"고 경고한다.
- 잘못 설계된 루프는 **토큰을 낭비하고, 영원히 돌고, 진행을 환각**한다. 루프의 품질은 모델이 아니라 **루프 설계**에서 갈린다.
- 토큰 비용 구조(token rich vs poor)에 따라 적용 가능성이 크게 달라진다. → 4장 가드레일 참조.
