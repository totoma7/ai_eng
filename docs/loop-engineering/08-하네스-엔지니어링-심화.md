# 8장 (심화). 하네스 엔지니어링 — AI 컨트롤 플레인

> [6장](06-하네스와-루프.md)에서 개요를 다뤘다. 이 장은 2026년 들어 공식 분야로 자리잡은
> 하네스 엔지니어링을 프로덕션 관점에서 깊이 파본다.

## 8.1 공식: Agent = Model + Harness

> **에이전트 = 모델 + 하네스.**
> 모델은 무상태(stateless) 토큰 예측기일 뿐이고, 하네스가 추론을 "실행 가능하고 신뢰할 수 있는 것"으로 만든다.

핵심 운영 원칙:

> "에이전트가 실수를 발견할 때마다, **같은 실수를 두 번 다시 못 하도록 엔지니어링**하라." — 대부분 하네스 개선으로 해결된다.

## 8.2 왜 2026년의 중심 투자처인가 — 구속 제약 논제

**Binding-Constraint Thesis**: 2025년부터 에이전트 신뢰성의 병목은 **모델 품질이 아니라 인프라(하네스) 품질**이다.

근거 데이터:

| 데이터 포인트 | 내용 |
|--------------|------|
| 기업 실패율 | 기업 AI 에이전트 프로젝트의 **최대 88%가 프로덕션 도달 실패** |
| 실패 원인 | 기업 AI 실패의 **65%가 하네스 수준 결함**으로 추적 (모델 추론 한계가 아님) |
| 하네스 효과 | 동일 모델 가중치라도 하네스에 따라 SWE-Bench 점수가 **10~20점** 출렁임 |
| 실증 사례 | LangChain 코딩 에이전트가 **모델 변경 없이** 하네스 최적화만으로 Terminal-Bench 30위 → 5위 |
| OpenAI 사례 | 2026년 2월, 소규모 팀이 5개월간 프로덕션 코드를 직접 쓰지 않고 **약 100만 줄** 내부 제품 산출 |

## 8.3 프로덕션 하네스의 5계층

| 계층 | 역할 | 실패 시 증상 |
|------|------|------------|
| ① 도구 오케스트레이션 | 도구 선택·체이닝·실행·동적 오류 복구 | 깨지기 쉬운 스크립트, 도구 오선택 |
| ② 검증 루프 | 실행 **중간**의 자동 QA (단위 테스트, 자기 비판) — 완료 후가 아니라 | 오류 복리화(compounding) |
| ③ 컨텍스트 & 메모리 | 코드베이스 인덱싱, 세션 간 이력 영속화 | 매번 같은 설명 반복, 컨벤션 위반 |
| ④ 가드레일 | 경계 제한, 보안 샌드박스, 예산 상한, 승인 게이트 | 폭주 비용, 파괴적 행동 |
| ⑤ 관측성 | 텔레메트리, 실행 추적, 감사 로그 | 디버깅 불가능, 회귀 검증 불가 |

6장의 7-plane 프레임워크와 겹치되, 이쪽은 **투자 우선순위 결정용** 분류다.
"어느 계층에 다음 투자를 할 것인가"를 메트릭으로 판단한다 (8.6 참조).

## 8.4 하네스가 고치는 모델 고유의 실패 패턴

| 실패 패턴 | 증상 | 하네스 해법 |
|----------|------|-----------|
| **승리 선언 편향** (victory declaration bias) | 결과 검증 없이 "완료" 선언 | 검증 루프 + 평가자 분리 (2장) |
| **컨텍스트 불안** (context anxiety) | 컨텍스트 창이 차면 서두르고 모서리를 깎음 | Ralph Loop — 원래 의도를 깨끗한 컨텍스트에 재주입 (3장) |
| **원샷 과욕** (one-shotting overreach) | 전체 문제를 한 번에 풀려다 문서화 안 된 엉킴 생성 | 단계 분해 + 커밋 크기 단위 강제 |

## 8.5 고급 하네스 패턴

- **Prefix Stability / KV-캐시 지역성**: 시스템 지시·이력에서 동적 요소(초 단위 타임스탬프 등)를 제거해 캐시 무효화 방지 → 토큰 비용 $3.00/MTok → **$0.30/MTok**, 지연 4배 감소. 모델 변경 없이.
- **Context Rot 방지**: 거대한 도구 출력이 누적되며 어텐션이 열화 → **가상 파일시스템**으로 메모리를 외부화 (4장 외부 메모리의 하네스 구현)
- **Logits Masking / 동적 도구 공급**: 작업 단계에 맞는 도구만 노출 — 과적재 프롬프트의 어텐션 분산 방지
- **Reasoning Sandwich**: 고성능 모델이 계획·검증, 저비용 모델이 중간 작업 — 오류 누적 방지를 위한 수학적 필연 (멀티 티어 모델 라우팅)

## 8.6 메트릭 — 무엇을 측정할 것인가

단계적 측정 접근 (Faros AI 프레임워크):

| 단계 | 전제 | 메트릭 예시 |
|------|------|-----------|
| Stage 1 (즉시) | 기존 데이터만으로 | 머지된 PR당 비용, 개발자당 컴퓨트 지출, 에이전트 PR의 time-to-merge, 코드 churn |
| Stage 2 | 세션↔PR 연결 인프라 | 첫 시도 성공률, **에이전트 PR 생존율**(머지 30일 후 코드 잔존), 결함 유출률 |
| Stage 3 | 분류/설문 추가 | 과업 복잡도 분포, 리뷰어 피로도·신뢰 신호 |

> "세션을 PR에 연결하는 인프라가 측정에서 **가장 어렵고 가장 가치 있는** 조각이다."

진단 휴리스틱: 과업 실패↑ → 하네스 문제(컨텍스트 불완전, 검증 누락) / AI 비용↑ → 토큰 낭비(중복 호출) / 개발자 신뢰↓ → 하네스가 에이전트의 추론을 리뷰어에게 전달하지 못함.

## 8.7 표준화 — MCP와 A2A

- **MCP (Model Context Protocol)**: 수직 방향 — 에이전트↔도구. 클라이언트 로직과 도구 구현 분리.
- **A2A (Agent-to-Agent)**: 수평 방향 — 에이전트↔에이전트 위임. 암호학적 "Agent Card"로 발견·위임.

둘 다 "AI의 USB-C" — 벤더 종속을 막는 **조달 레버리지**.

전략: **상용 인프라는 사고**(관리형 런타임, 텔레메트리), **고유 자산은 만든다**(도메인 도구, 커스텀 평가 데이터셋, 환경 맵).

## 8.8 "시니어 엔지니어 세금" 문제

AI 생성 코드는 표면 품질이 높지만 미묘한 오류를 품는다 → 리뷰어가 "그럴듯하게 쓰인 코드"에서 결함을 사냥해야 함 → 리뷰 피로 → 실수 유출률 증가. 5장의 **이해 부채**가 조직 레벨에서 나타난 형태다. 2026 AI Engineering Report: 처리량은 늘었지만 **버그·인시던트·재작업이 더 빠르게 증가** — 하네스 설계 엄밀성이 그 격차를 메운다.
