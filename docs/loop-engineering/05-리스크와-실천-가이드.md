# 5장. 리스크와 실천 가이드

## 5.1 세 가지 본질적 리스크

### ① 검증은 끝까지 당신 책임
> "방치된 루프는 방치된 채로 실수한다. 'Done'은 **주장**이지 **증명**이 아니다."

루프가 "완료"라고 말하는 것과 실제로 완료된 것은 다르다. 자동 검증 게이트가 있어도, 게이트가 검사하지 않는 영역은 사람이 본다.

### ② 이해 부채 (Comprehension Debt)
- 코드가 **이해 속도보다 빠르게** 출하되면, '존재하는 것'과 '파악하고 있는 것' 사이의 격차가 벌어진다.
- 사례: 25시간, 1,300만 토큰, 3만 줄 — 이 코드를 누가 이해하는가?
- 완화책: **루프 출력을 읽는 것은 타협 불가**. 계획 승인, PR 사인오프 같은 **사람 결정 지점**을 머지 전에 유지.

### ③ 인지적 항복 (Cognitive Surrender)
> "동일한 루프를 가진 두 엔지니어가 정반대의 결과를 얻는다 — 한 명은 일을 깊이 이해하고, 다른 한 명은 생각을 회피한다. 루프 자체는 어느 쪽인지 모른다."

편안한 자동화는 판단의 포기를 유혹한다. Osmani의 결론:

> "루프를 만들어라. 단, **엔지니어로 남을 사람**처럼 만들어라. '실행 버튼만 누르는 사람'이 아니라."

## 5.2 핵심 멘탈 모델 정리

1. **루프 = 트리거 + 검증 가능한 목표.** 둘 중 하나라도 약하면 루프가 아니라 토큰 소각로다.
2. **레버리지는 프롬프트가 아니라 루프 설계에 있다.** 에이전트 품질 = 루프 품질.
3. **피드백 없는 루프는 자기 동의일 뿐.** 테스트/타입/리뷰 게이트가 신뢰의 원천.
4. **검증자를 분리하라.** 작업자가 자기 완료를 선언하게 두지 마라.
5. **자율성은 없애는 게 아니라 경계 짓는 것.** 예산, 상한, 체크포인트.
6. **루프 설계는 프롬프트 엔지니어링보다 어렵다.** 일이 쉬워진 게 아니라 레버리지 지점이 이동했을 뿐.

## 5.3 시작하는 사람을 위한 단계별 로드맵

| 단계 | 실천 | 익히는 것 |
|------|------|----------|
| 1 | 에이전트(Claude Code 등)로 단일 과업을 주고 3단계 루프(수집→실행→검증)를 **관찰** | 루프의 기본 리듬 |
| 2 | 검증 게이트 추가: "테스트가 통과할 때까지 고쳐라" 식의 **검증 가능한 목표** 명세 | 종료 조건 설계 |
| 3 | `/goal`·`/loop` 또는 셸 스크립트로 **단순 반복 루프**(ralph식) 구성 | 트리거 + 외부 메모리 |
| 4 | 작업자/리뷰어 **서브에이전트 분리** | 검증 분리 |
| 5 | worktree 병렬화 + MCP 커넥터로 **오케스트레이션 루프** 구축 | 멀티 에이전트 감독 |
| 6 | 예산·상한·무진행 감지·휴먼 체크포인트 계측 | 프로덕션 가드레일 |

> 🛠 **2~3단계 실습 코드**: [practice/ralph-loop/](../../practice/ralph-loop/README.md) —
> 검증 가능한 목표 명세(goal.md)와 ralph식 루프(ralph.sh)를 60줄 셸 스크립트로 직접 구현한 예제.
> mock 에이전트로 수렴 과정과 가드레일(무진행 감지, 반복 상한) 동작을 재현할 수 있다.

## 5.4 다음 학습 주제

- **하네스 엔지니어링 (Harness Engineering)**: 컨텍스트 엔지니어링의 다음 진화로 불림. 루프 정책, 도구 표면, 컨텍스트, 샌드박스, 멀티 에이전트 라우팅, 관측성, 모델 라우팅의 7개 평면(seven-plane) 프레임워크.
- **멀티 에이전트 감독(supervision)**: 단일 루프 다음 레이어.
- **평가(Evals) 설계**: 검증 게이트의 품질이 곧 루프의 품질이므로.
